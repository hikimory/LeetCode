## Шаг 1: С чего начать обдумывание? (Анализ ограничений и структуры)

Первое, на что всегда нужно смотреть — это свойства графа и ограничения ($n \le 10^5$).

1. Что за граф перед нами? В условии сказано: "ориентированный граф, который стал бы деревом, если бы ребра были двунаправленными". Это ключевая подсказка. Перед нами ориентированное дерево (полидерево).
    
    * Свойство дерева: Между любыми двумя вершинами существует ровно один простой путь (если не учитывать направление стрелок).

2. Что означает «разворот ребра»? Если нам нужно пройти от вершины $A$ к вершине $B$, а стрелка стоит как $A \leftarrow B$, нам придется ее развернуть. Это будет стоить нам +1 к цене пути. Если стрелка уже стоит как $A \rightarrow B$, то цена прохода равна 0.

3. Оценка «в лоб» (Брутфорс): Мы могли бы запустить обычный обход (DFS или BFS) из каждой вершины по очереди и посчитать, сколько ребер нужно развернуть, чтобы дойти до всех остальных.

* Время одного обхода: $O(n)$.

* Вершин всего: $n$.

* Итоговое время: $O(n^2)$.

* При $n = 10^5$ алгоритм за $O(n^2)$ выдаст Time Limit Exceeded (TLE). Нам нужно решение за $O(n)$.

## Шаг 2: Как прийти к логическому решению? (Ищем закономерности)

Давай проведем мысленный эксперимент. Представь, что мы уже посчитали ответ для какой-то одной вершины — пусть это будет Корень (Root). Мы знаем, сколько разворотов нужно, чтобы из Корня добраться во все стороны.

Теперь мы хотим узнать ответ для соседней вершины Соседа (Neighbor), которая соединена с Корнем ребром. Чем их пути ко всем остальным вершинам отличаются?

Поскольку это дерево, удаление ребра между Root и Neighbor делит граф на две изолированные части:

1. Поддерево со стороны Root.

2. Поддерево со стороны Neighbor.

Когда мы мысленно «переносим» центр управления из Root в Neighbor:

* Для всех вершин в поддереве Neighbor путь теперь стал на один шаг короче (ведь нам больше не нужно переходить из Root в Neighbor).

* Для всех вершин в поддереве Root путь стал на один шаг длиннее (теперь нам нужно сначала перейти из Neighbor в Root).

А теперь самое главное: как изменится количество разворотов? Изменится только состояние ребра между Root и Neighbor!

* Если исходное ребро было направлено как Root -> Neighbor: для Корня оно было «бесплатным» (0), а для Соседа станет «платным» (+1), потому что теперь ему нужно идти против шерсти (Neighbor -> Root). Значит: $\text{Ans(Neighbor)} = \text{Ans(Root)} + 1$.

* Если исходное ребро было направлено как Root <- Neighbor: для Корня оно требовало разворота (+1), а для Соседа оно уже направлено куда надо (0). Значит: $\text{Ans(Neighbor)} = \text{Ans(Root)} - 1$.

Зная ответ для одной вершины, мы можем за $O(1)$ узнать ответ для любого её соседа, просто посмотрев на направление ребра между ними!

Этот паттерн в динамическом программировании называется Rerooting DP (ДП на деревьях со сменой корня) или Double DFS.

## Шаг 3: Способы реализации алгоритмаОптимальный алгоритм всегда будет состоять из двух фаз (двух обходов графа):

1. Первый проход (Сбор базовой информации):Выбираем абсолютно любую вершину (например, 0) в качестве временного корня. Запускаем DFS, который считает, сколько разворотов нужно, чтобы добраться из вершины 0 во все остальные вершины графа. Это дает нам правильный ответ для answer[0].

2. Второй проход (Пересчет для остальных — Rerooting):Запускаем второй DFS из вершины 0. Идя по ребрам к соседям, мы используем выведенное нами правило: ans[neighbor] = ans[root] + 1 или - 1 в зависимости от направления стрелки. Таким образом, мы заполняем весь массив answer за один обход.

## Пример 1

У нас есть 4 вершины (0, 1, 2, 3) и следующие ориентированные ребра:2 -> 0, 2 -> 1, 1 -> 3

## Шаг 1: Построение двунаправленного графа

Мы превращаем граф в неориентированный, но для каждого направления запоминаем, «платное» оно или нет. Если мы идем по направлению стрелки — цена 0 (разворот не нужен). Если идем против стрелки — цена 1 (нужно развернуть).

Вот как выглядит наш список смежности для каждой вершины (сосед, цена):

* 0: [(2, 1)] (чтобы пройти в 2, нужно развернуть ребро 2->0)

* 1: [(2, 1), (3, 0)] (в 2 идти против стрелки = 1; в 3 идти по стрелке = 0)

* 2: [(0, 0), (1, 0)] (в 0 и 1 идем по стрелкам = 0)

* 3: [(1, 1)] (чтобы пройти в 1, нужно развернуть 1->3)

## Шаг 2: Первый проход (DFS1) — Считаем ответ для корня 0

Мы запускаем обход из вершины 0 и смотрим, сколько разворотов нужно, чтобы волна дошла до всех остальных вершин.

* Из 0 мы можем пойти только в 2. Ребро 2 -> 0 направлено против нашего движения, поэтому цена = 1.

* Из 2 мы можем пойти в 1. Ребро 2 -> 1 направлено по нашему движению, цена = 0.

* Из 1 мы можем пойти в 3. Ребро 1 -> 3 направлено по нашему движению, цена = 0.

Складываем все стоимости: $1 + 0 + 0 = 1$.
Таким образом, для корня 0 нам нужен всего 1 разворот.
Записываем в массив ответов: ans[0] = 1.

## Шаг 3: Второй проход (DFS2) — Перевешивание (Rerooting)

Теперь мы начинаем двигаться от корня 0 по дереву вниз и динамически пересчитывать ответы для остальных вершин за $O(1)$ на каждый шаг, не запуская поиск заново.

### Переход 0 -> 2:

Мы стоим в 0 (где ans[0] = 1) и хотим узнать ответ для 2.

Смотрим на оригинальное ребро между ними. Оригинальное ребро — это 2 -> 0.

Логика: Поскольку в реальности стрелка уже смотрит на 0, то для вершины 2 это ребро является идеальным «выходом» наружу к вершине 0. Значит, вершине 2 не нужно тратить разворот на это ребро, в отличие от вершины 0.

Математика: Так как ребро направлено от 2 к 0, при переносе корня в 2 стоимость уменьшается:

$$\text{ans}[2] = \text{ans}[0] - 1 = 1 - 1 = 0$$

### Переход 2 -> 1:

Мы стоим в 2 (где ans[2] = 0) и переходим в 1.

Оригинальное ребро — 2 -> 1.

* Логика: Стрелка изначально направлена от 2 к 1. Для вершины 2 это было удобно. Но теперь конем становится 1, и чтобы из 1 добраться обратно в 2 (и во все её поддеревья), ребро 2 -> 1 придется развернуть. Значит, стоимость увеличивается на 1.

* Математика:

$$\text{ans}[1] = \text{ans}[2] + 1 = 0 + 1 = 1$$

### Переход 1 -> 3:

Мы стоим в 1 (где ans[1] = 1) и переходим в 3.

Оригинальное ребро — 1 -> 3.

* Логика: Стрелка смотрит от 1 к 3. Если новым корнем становится 3, то чтобы из неё попасть в 1 (и пойти дальше по графу), ребро 1 -> 3 придется развернуть. Стоимость снова увеличивается на 1.

* Математика:
 
$$\text{ans}[3] = \text{ans}[1] + 1 = 1 + 1 = 2$$

## Итог

Пройдя по дереву один раз вниз, мы заполнили весь массив:ans = [1, 1, 0, 2]
