## Как добиться $O(1)$?

Если использовать только одну структуру данных, условие выполнить не получится:

* Обычный словарь (Dictionary) дает быстрый доступ за $O(1)$, но в нем нет встроенного порядка элементов — мы не узнаем, какой элемент использовался реже всего.

* Список (List) хранит порядок, но поиск или удаление элемента из середины занимает $O(n)$.

Решение: Объединить две структуры данных:

1. Двусвязный список (Doubly Linked List): в нем будут храниться ключи и значения. Тот элемент, к которому обратились последним, мы будем перемещать в голову (head) списка. Самый «старый» элемент всегда будет оказываться в хвосте (tail). В двусвязном списке удаление и вставка элемента (если у нас уже есть на него ссылка) занимают $O(1)$.

2. Хэш-таблица (Dictionary<int, LinkedListNode>): ключами будут key из задачи, а значениями — ссылки на конкретные узлы нашего двусвязного списка. Это позволит мгновенно находить нужный узел в списке за $O(1)$.

## Пример:
Представь, что мы создаем кэш с емкостью capacity = 2.

В самом начале, сразу после вызова new LRUCache(2), у нас создаются два пустых фиктивных узла — Голова (H) и Хвост (T). Они связаны друг с другом напрямую, а словарь _cache пока пуст.

```
Plaintext

Список:    [ H ] <---> [ T ]
Словарь:   { }
```

### Шаг 1: put(1, 1)

Мы хотим добавить ключ 1 со значением 1.

1. Программа смотрит в словарь — ключа 1 там нет.

2. Создается новый узел (Key: 1, Value: 1).

3. Этот узел добавляется в словарь: { 1: узел_1 }.

4. Вызывается метод AddToHead. Новый узел вклинивается сразу после Головы:

```
Plaintext

Список:    [ H ] <---> [ 1:1 ] <---> [ T ]
Словарь:   { 1: узел_1 }
```

### Шаг 2: put(2, 2)

Добавляем ключ 2 со значением 2.

1. Ключа 2 в словаре нет. Создаем узел (Key: 2, Value: 2).

2. Добавляем в словарь: { 1: узел_1, 2: узел_2 }.

3. Снова вызываем AddToHead. Новый узел встает между Головой и узлом 1:

```
Plaintext

Список:    [ H ] <---> [ 2:2 ] <---> [ 1:1 ] <---> [ T ]
Словарь:   { 1: узел_1, 2: узел_2 }
```

Обрати внимание: узел [ 1:1 ] сместился ближе к Хвосту. Теперь именно он — кандидат на удаление, если место закончится.

### Шаг 3: get(1)

Мы запрашиваем элемент с ключом 1.

1. Программа проверяет словарь. Ключ 1 там есть! Мы мгновенно получаем ссылку на узел_1 за $O(1)$.

2. Так как к элементу 1 только что обратились, он теперь «самый свежий». Код вызывает метод MoveToHead(узел_1).

3. Внутри MoveToHead происходят два действия:
   * RemoveNode: узел 1 аккуратно «вырезается» из своего текущего места (соседи [ 2:2 ] и [ T ] связываются напрямую друг с другом).
  
   * AddToHead: узел 1 вставляется в самое начало, сразу после Головы. 

```
Plaintext

Список:    [ H ] <---> [ 1:1 ] <---> [ 2:2 ] <---> [ T ]
Словарь:   { 1: узел_1, 2: узел_2 }
```
Результат: Местами в списке поменялись, теперь самым «старым» (ближним к Хвосту) стал узел [ 2:2 ]. Метод возвращает значение 1.

### Шаг 4: put(3, 3) — Переполнение!

Пытаемся добавить ключ 3.

1. Ключа 3 в словаре нет. Создаем новый узел (Key: 3, Value: 3).
   
2. Добавляем в словарь и вставляем в голову списка:

```
Plaintext

Список:    [ H ] <---> [ 3:3 ] <---> [ 1:1 ] <---> [ 2:2 ] <---> [ T ]
Словарь:   { 1: узел_1, 2: узел_2, 3: узел_3 }
```

3. Проверка емкости: Сейчас в словаре 3 элемента, а лимит (_capacity) равен 2. Пора выселять наименее используемый (LRU).

4. Код смотрит на узел прямо перед Хвостом: Node lruNode = _tail.Prev; — это узел [ 2:2 ].

5. Вызывается RemoveNode(lruNode), который выкидывает этот узел из двусвязного списка.

6. Вызывается _cache.Remove(lruNode.Key);, который удаляет ключ 2 из словаря.

```
Plaintext

Список:    [ H ] <---> [ 3:3 ] <---> [ 1:1 ] <---> [ T ]
Словарь:   { 1: узел_1, 3: узел_3 }
```

Элемент 2 успешно стерт из памяти за константное время.

### Шаг 5: get(2)

1. Программа ищет ключ 2 в словаре.

2. Его там нет (мы только что его удалили).

3. Метод возвращает -1. 

## Почему это работает так быстро?
Благодаря двусвязному списку, чтобы переставить узел из середины в начало, нам не нужно сдвигать все остальные элементы в массиве (как пришлось бы делать в List). Мы просто меняем 4 указателя (Next и Prev у самого узла и его соседей). А благодаря словарю мы находим этот узел в памяти за один шаг, не перебирая весь список с начала до конца.
